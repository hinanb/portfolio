Hardcoding credentials in source code is one of the most common and dangerous security mistakes in software development. This article demonstrates how to avoid it using Azure DevOps.

## Key Practices

- Use **Azure Key Vault** to store secrets securely
- Reference secrets in pipeline variables instead of hardcoding
- Use **service connections** for authenticated access to external services
- Enable **secret scanning** in your CI/CD pipeline
- Rotate credentials regularly and audit access

## Azure DevOps Implementation

Configure variable groups linked to Key Vault, reference them in your pipeline YAML, and never commit `.env` files or connection strings to version control.

```yaml
variables:
  - group: my-keyvault-secrets

steps:
  - script: echo "Using secure variables from Key Vault"
    env:
      DB_CONNECTION: $(DatabaseConnectionString)
```

## Takeaway

Treat credentials as infrastructure, not code. Every secret should live in a vault, every pipeline should reference it securely, and every developer should understand why hardcoding is never acceptable.
